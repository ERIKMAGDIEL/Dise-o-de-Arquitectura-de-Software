Resumen

Este documento describe la estructura técnica para la “Aplicación de Gestión de Dormitorio Estudiantil", identificando sus componentes principales, cómo interactúan entre sí y los patrones de diseño que guiarán su construcción.

1. Diagrama de Arquitectura

El sistema se basará en una arquitectura cliente-servidor de tres capas distribuida. Esta arquitectura desacopla la presentación, la lógica de negocio y el almacenamiento de datos, lo que resulta en un sistema más modular, escalable y fácil de mantener.

Capa de Cliente (Presentación): Es la aplicación móvil desarrollada en Flutter. Se encarga exclusivamente de la interfaz de usuario (UI) y la experiencia de usuario (UX). Se comunica con el backend a través de una API REST.

Capa de Servidor (Lógica de Negocio): Es el backend desarrollado en Node.js. Contiene toda la lógica de negocio, procesa las solicitudes del cliente, valida datos, se comunica con la base de datos y expone los datos a través de una API RESTful segura.

Capa de Datos (Persistencia): Es la base de datos (SQL). Su única responsabilidad es almacenar y recuperar datos de manera eficiente y segura. No contiene ninguna lógica de negocio.

2. Descripción de Componentes y sus Interfaces

2.1 Componente: Aplicación Móvil (Cliente - Flutter)

Descripción: Es el punto de entrada para todos los usuarios (estudiantes y administradores). Es una aplicación multi-plataforma que se enfoca en presentar los datos de manera intuitiva.

Sub-módulos:

  • Módulo de Vistas (UI): Pantallas para login, dashboard, anuncios, formulario de permisos, lista de asistencia, etc.
  • Gestor de Estado (State Management): Controla el estado de la aplicación para mantener la UI consistente (ej. usando Provider o BLoC).
  • Cliente HTTP: Módulo responsable de realizar las llamadas a la API del backend y manejar las respuestas.

La aplicación es responsable de comunicar exclusivamente con el backend a través de su API RESTful. Consume los endpoints expuestos por el servidor para enviar y recibir datos en formato JSON.

2.3 Sub-módulos (Capas)

El servidor se organiza en capas que separan responsabilidades y favorecen la escalabilidad del sistema.

Capa Descripción
Capa de Ruteo Define los endpoints principales de la API (ej. /api/auth, /api/detallecuarto, /api/monitores).
Capa de Controladores Recibe las peticiones, las valida y llama a los servicios correspondientes.
Capa de Servicios Implementa la lógica de negocio principal (ej. la lógica para aprobar un permiso).
Capa de Acceso a Datos (DAL) Abstrae la comunicación con la base de datos, garantizando seguridad y consistencia.

API RESTFUL: Expone una interfaz a través de HTTP para ser consumida por el cliente. Su responsabilidad es manejar las acciones (GET, POST, PUT, DELETE) sobre los recursos del sistema.

2.5 Componente: Base de Datos

Descripción: Es el sistema de almacenamiento persistente de la aplicación, implementado en SQL Server. Su responsabilidad es guardar, organizar y recuperar toda la información de manera eficiente, segura y consistente. Alberga las entidades principales del dominio, tales como:

  • Catálogos: Tablas auxiliares para definir Tipos de Culto, Niveles de Amonestación, Criterios de Limpieza, Roles, etc.
  • Usuarios: Información de autenticación y roles (Estudiante, Monitor, Preceptor).
  • Perfiles: Datos específicos de Estudiantes (incluyendo carrera, cuarto, pasillo si es monitor) y Preceptores.
  • Estructura: Información sobre Cuartos y Pasillos.
  • Registros de Actividad: Datos sobre Limpieza (evaluaciones y detalles), Reportes disciplinarios, Amonestaciones y Asistencia a Cultos/Actividades.

La base de datos no expone una interfaz directa a la aplicación cliente. Su única interfaz es a través de la Capa de Acceso a Datos (DAL) del backend (Node.js/Express). El DAL interactúa con la base de datos utilizando el driver mssql para Node.js, ejecutando consultas SQL parametrizadas para realizar operaciones CRUD (Crear, Leer, Actualizar, Borrar) y asegurar la integridad de los datos mediante el uso de llaves primarias, foráneas y transacciones donde sea necesario.

3. Patrones de Diseño

3.1 Patrones Generales / Arquitectónicos

  • Arquitectura Cliente-Servidor: Es el patrón fundamental. La aplicación Flutter (cliente) se comunica con la API Node.js (servidor) para obtener y modificar datos.
  • API RESTful: El backend expone sus funcionalidades siguiendo los principios REST (Representational State Transfer) a través de endpoints HTTP (GET, POST, PUT), utilizando JSON como formato de datos.
  • Arquitectura por Capas (Layered Architecture): Tanto el frontend como el backend están organizados en capas lógicas con responsabilidades distintas:
    • Flutter: Presentación (UI) → Lógica/Estado (Widgets, Provider) → Servicios → Modelos.
    • Node.js: Ruteo → Controladores/Manejadores → Acceso a Datos (DAL).

3.2 Patrones Específicos del Frontend (Flutter)

  • Provider Pattern (para State Management): Utilizamos el paquete provider para gestionar el estado global de la aplicación (como la información del usuario logueado con UserProvider y el tema con ThemeProvider) y hacerlo accesible a diferentes partes del árbol de widgets.
  • Service Layer / Repository Pattern (Simplificado): Las clases Service.dart (ej. AuthService, LimpiezaService) encapsulan la lógica de comunicación con la API. Actúan como una capa de abstracción entre la UI y las llamadas HTTP, similar a un Repositorio o un Servicio específico para cada entidad o grupo de endpoints.
  • Model Class / Data Transfer Object (DTO): Las clases model.dart (ej. Reporte, Amonestacion) definen la estructura de los datos que se transfieren entre la API y la aplicación, facilitando el manejo de la información.

3.3 Patrones Específicos del Backend (Node.js/Express)

  • Middleware Pattern: Express utiliza extensamente middlewares (cors(), express.json(), etc.) para procesar las peticiones HTTP en una cadena de funciones antes de llegar al manejador final de la ruta.
  • Routing Pattern: El sistema express.Router() implementa este patrón para mapear las URLs y métodos HTTP a las funciones controladoras correspondientes (async (req, res) => {...}).
  • Controller Pattern: Las funciones async (req, res) => {...} asociadas a cada ruta actúan como controladores, recibiendo la petición, interactuando con las capas inferiores (lógica, datos) y formulando la respuesta.
  • Transaction Pattern: Lo usamos explícitamente en la ruta POST /api/limpieza/registrar para asegurar que la inserción en Limpieza y LimpiezaDetalle sea atómica (o todo tiene éxito, o todo se revierte).

4. Diagrama de Secuencia

4.1 Lista de Secuencias de Eventos Principales

Esta lista describe las interacciones paso a paso para las funcionalidades clave de la aplicación de gestión de dormitorios.

Actores:

  • Usuario: Cualquier persona antes de iniciar sesión.
  • Estudiante: Usuario con Rol 3.
  • Monitor: Usuario con Rol 2 (también es Estudiante).
  • Preceptor: Usuario con Rol 1.
  • App Móvil: La aplicación Flutter.
  • API Backend: La API Node.js/Express.
  • Base de Datos (BD): SQL Server.

4.2 Autenticación (Todos los Roles)

4.2.1 Secuencia: Inicio de Sesión Exitoso

  1. Usuario → App Móvil: Ingresa UsuarioID (Matrícula/Clave) y Contraseña, presiona "Acceder".
  2. App Móvil → API Backend: POST /api/auth/login con credenciales.
  3. API Backend → Base de Datos: SELECT en Usuarios buscando coincidencia y LEFT JOIN con Estudiantes/Preceptores para obtener datos.
  4. Base de Datos → API Backend: Devuelve datos del usuario (incluyendo IdRol, NombreCompleto, etc.).
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: {...} }.
  6. App Móvil → App Móvil: Guarda datos del usuario en UserProvider.
  7. App Móvil → App Móvil: Lee IdRol del UserProvider.
  8. App Móvil → Usuario: Navega al Dashboard correspondiente (Estudiante, Monitor o Preceptor).

4.2.2 Secuencia: Inicio de Sesión Fallido (Credenciales Inválidas)

  1. Usuario → App Móvil: Ingresa UsuarioID y Contraseña, presiona "Acceder".
  2. App Móvil → API Backend: POST /api/auth/login con credenciales.
  3. API Backend → Base de Datos: SELECT en Usuarios.
  4. Base de Datos → API Backend: No devuelve resultados.
  5. API Backend → App Móvil: Respuesta 401 Unauthorized con { success: false, message: '...' }.
  6. App Móvil → Usuario: Muestra mensaje de error "Credenciales inválidas".

4.3 Funcionalidades del Estudiante

4.3.1 Secuencia: Ver Propia Calificación de Limpieza

  1. Estudiante → App Móvil: Navega a la pantalla "Limpieza".
  2. App Móvil → App Móvil: Lee idCuarto del UserProvider.
  3. App Móvil → API Backend: GET /api/limpieza/detalle/{idCuarto}.
  4. API Backend → Base de Datos: SELECT TOP 1 en Limpieza con JOIN a LimpiezaDetalle y CriteriosLimpieza para el idCuarto dado.
  5. Base de Datos → API Backend: Devuelve los detalles de la última limpieza (o null).
  6. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: {...} } (o data: null).
  7. App Móvil → Estudiante: Muestra la calificación detallada (o mensaje "No hay registros").

4.3.2 Secuencia: Ver Propios Reportes

  1. Estudiante → App Móvil: Navega a la pantalla "Mis Reportes".
  2. App Móvil → App Móvil: Lee matrícula del UserProvider.
  3. App Móvil → API Backend: GET /api/reportes/estudiante/{matricula}.
  4. API Backend → Base de Datos: SELECT en Reportes con JOIN a Estudiantes/Preceptores (para nombre reportante) donde MatriculaReportado coincida.
  5. Base de Datos → API Backend: Devuelve la lista de reportes.
  6. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  7. App Móvil → Estudiante: Muestra la lista de sus reportes.

4.3.3 Secuencia: Escanear QR para Registrar Asistencia

  1. Estudiante → App Móvil: Navega a "Asistencia", presiona "Escanear QR".
  2. App Móvil → App Móvil: Activa la cámara.
  3. (Monitor muestra QR generado previamente con { "idTipoCulto": X, "registradoPor": "MatriculaMonitor" }).
  4. Estudiante → App Móvil: Escanea el QR.
  5. App Móvil → App Móvil: Lee y decodifica el JSON del QR. Lee matriculaEstudiante del UserProvider.
  6. App Móvil → API Backend: POST /api/asistencia/registrar con { matriculaEstudiante, idTipoCulto, registradoPor }.
  7. API Backend → Base de Datos: INSERT en AsistenciasCultos.
  8. Base de Datos → API Backend: Confirma inserción.
  9. API Backend → App Móvil: Respuesta 201 Created con { success: true, message: '...' }.
  10. App Móvil → Estudiante: Muestra mensaje de éxito "Asistencia registrada".

4.4 Funcionalidades del Monitor

4.4.1 Secuencia: Generar QR de Asistencia

  1. Monitor → App Móvil: Navega a "Generar QR".
  2. App Móvil → API Backend: GET /api/cultos/tipos (para llenar el dropdown).
  3. API Backend → Base de Datos: SELECT * FROM Cat_TipoCulto.
  4. Base de Datos → API Backend: Devuelve lista de tipos de culto.
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  6. App Móvil → Monitor: Muestra dropdown con tipos de culto.
  7. Monitor → App Móvil: Selecciona un tipo de culto (ej. "Matutino", idTipoCulto: 1), presiona "Generar QR".
  8. App Móvil → App Móvil: Lee matrícula del monitor desde UserProvider.
  9. App Móvil → App Móvil: Crea el JSON: { "idTipoCulto": 1, "registradoPor": "matriculaMonitor" }.
  10. App Móvil → Monitor: Genera y muestra la imagen del QR correspondiente a ese JSON.

4.4.2 Secuencia: Ver Lista de Asistencias de un Culto/Fecha

  1. Monitor → App Móvil: Navega a "Ver Asistencias".
  2. App Móvil → API Backend: GET /api/cultos/tipos (para dropdown de culto).
  3. (Pasos 3–5 como en Generar QR).
  4. App Móvil → Monitor: Muestra dropdown de culto y selector de fecha.
  5. Monitor → App Móvil: Selecciona culto (ej. idTipoCulto: 1) y fecha (ej. 2025-10-20).
  6. App Móvil → API Backend: GET /api/asistencia/culto?idTipoCulto=1&fecha=2025-10-20.
  7. API Backend → Base de Datos: SELECT en AsistenciasCultos con JOIN a Estudiantes filtrando por IdTipoCulto y Fecha.
  8. Base de Datos → API Backend: Devuelve lista de asistentes (Matrícula, NombreCompleto).
  9. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  10. App Móvil → Monitor: Muestra la lista de estudiantes asistentes.

4.4.3 Secuencia: Ver Lista de Cuartos para Evaluar Limpieza

  1. Monitor → App Móvil: Navega a "Limpieza".
  2. App Móvil → API Backend: GET /api/limpieza/cuartos-con-calificacion.
  3. API Backend → Base de Datos: SELECT en Cuartos con LEFT JOIN a la última Limpieza de cada cuarto.
  4. Base de Datos → API Backend: Devuelve lista de cuartos (IdCuarto, NumeroCuarto, IdPasillo, UltimaCalificacion).
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  6. App Móvil → App Móvil: Agrupa los cuartos por IdPasillo.
  7. App Móvil → Monitor: Muestra la lista de cuartos agrupada por pasillo.

4.4.4 Secuencia: Registrar Evaluación de Limpieza

  1. Monitor → App Móvil: En "Limpieza", presiona el ícono de escoba en un cuarto (ej. IdCuarto: 1).
  2. App Móvil → App Móvil: Navega a RegistrarLimpiezaScreen, pasando idCuarto: 1.
  3. App Móvil (RegistrarLimpiezaScreen) → API Backend: GET /api/limpieza/criterios.
  4. API Backend → Base de Datos: SELECT * FROM CriteriosLimpieza.
  5. Base de Datos → API Backend: Devuelve lista de criterios.
  6. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  7. App Móvil → Monitor: Muestra la lista de criterios con selectores de calificación (1–10) y campos adicionales.
  8. Monitor → App Móvil: Ingresa todas las calificaciones y observaciones, presiona "Guardar".
  9. App Móvil → App Móvil: Lee matrícula del monitor desde UserProvider.
  10. App Móvil → API Backend: POST /api/limpieza/registrar con { idCuarto: 1, evaluadoPor: matriculaMonitor, detallesMatutinos: [...], ... }.
  11. API Backend → Base de Datos: Inicia transacción. INSERT en Limpieza. INSERT en LimpiezaDetalle. COMMIT transacción.
  12. Base de Datos → API Backend: Confirma inserción.
  13. API Backend → App Móvil: Respuesta 201 Created con { success: true, message: '...', idLimpieza: ... }.
  14. App Móvil → Monitor: Muestra mensaje de éxito.
  15. App Móvil → App Móvil: Regresa a la pantalla anterior (limpieza_monitor_screen.dart), devolviendo true.
  16. App Móvil (limpieza_monitor_screen.dart) → App Móvil: Recarga la lista de cuartos.

4.4.5 Secuencia: Buscar Reportes de un Estudiante

  1. Monitor → App Móvil: Navega a "Reportes", ingresa matrícula, presiona "Buscar".
  2. App Móvil → API Backend: GET /api/reportes/estudiante/{matricula}.
  3. (Pasos 4–7 como en "Estudiante: Ver Propios Reportes", pero la App usa ReporteMonitor).
  4. App Móvil → Monitor: Muestra la lista de reportes para esa matrícula.

4.4.6 Secuencia: Crear Reporte para un Estudiante (desde Buscador)

  1. Monitor → App Móvil: En "Reportes", después de buscar una matrícula, presiona "+ Nuevo Reporte".
  2. App Móvil → App Móvil: Navega a CrearReporteScreen, pasando la matriculaEstudiante buscada.
  3. Monitor → App Móvil: Escribe el motivo, presiona "Guardar".
  4. App Móvil → App Móvil: Lee matrícula del monitor (reportadoPor) del UserProvider.
  5. App Móvil → API Backend: POST /api/reportes/crear con { matriculaReportado, reportadoPor, tipoUsuarioReportante: 'Monitor', motivo }.
  6. (Pasos 11–15 como en "Registrar Evaluación de Limpieza", pero para reportes).

4.5 Funcionalidades del Preceptor

4.5.1 Secuencia: Ver Lista de Monitores

  1. Preceptor → App Móvil: Navega a "Monitores".
  2. App Móvil → API Backend: GET /api/usuarios/monitores.
  3. API Backend → Base de Datos: SELECT U.UsuarioID, E.NombreCompleto FROM Usuarios U JOIN Estudiantes E ON U.UsuarioID = E.Matricula WHERE U.IdRol = 2.
  4. Base de Datos → API Backend: Devuelve lista de monitores.
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  6. App Móvil → Preceptor: Muestra la lista de monitores actuales.

4.5.2 Secuencia: Asignar Rol de Monitor a Estudiante

  1. Preceptor → App Móvil: En "Monitores", ingresa matrícula de estudiante, presiona "Asignar Rol".
  2. App Móvil → API Backend: PUT /api/usuarios/{matricula}/rol con { nuevoRol: 2 }.
  3. API Backend → Base de Datos: UPDATE Usuarios SET IdRol = 2 WHERE UsuarioID = {matricula}.
  4. Base de Datos → API Backend: Confirma actualización.
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, message: '...' }.
  6. App Móvil → Preceptor: Muestra mensaje de éxito.
  7. App Móvil → App Móvil: Recarga la lista de monitores.

4.5.3 Secuencia: Quitar Rol de Monitor

  1. Preceptor → App Móvil: En "Monitores", presiona el ícono de eliminar en un monitor.
  2. App Móvil → Preceptor: Muestra diálogo de confirmación.
  3. Preceptor → App Móvil: Confirma.
  4. App Móvil → API Backend: PUT /api/usuarios/{matricula}/rol con { nuevoRol: 3 }.
  5. API Backend → Base de Datos: UPDATE Usuarios SET IdRol = 3 WHERE UsuarioID = {matricula}. UPDATE Estudiantes SET IdPasillo = NULL, IdDormitorio = NULL WHERE Matricula = {matricula}.
  6. (Pasos 4–7 como en Asignar Rol).

4.5.4 Secuencia: Ver Todos los Reportes (con paginación/búsqueda)

  1. Preceptor → App Móvil: Navega a "Reportes".
  2. App Móvil → API Backend: GET /api/reportes?page=1&limit=20.
  3. API Backend → Base de Datos: SELECT ... FROM Reportes R JOIN Estudiantes E ... ORDER BY ... OFFSET 0 ROWS FETCH NEXT 20 ROWS ONLY. SELECT COUNT(*)....
  4. Base de Datos → API Backend: Devuelve los primeros 20 reportes y el conteo total.
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...], total: N, page: 1, limit: 20 }.
  6. App Móvil → Preceptor: Muestra los primeros 20 reportes.
  7. (Opcional: Si el preceptor escribe en buscar) Preceptor → App Móvil: Ingresa texto en buscador.
  8. App Móvil → API Backend: GET /api/reportes?page=1&limit=20&search={texto}.
  9. (Pasos 3–6 se repiten con el filtro WHERE).
  10. (Opcional: Si el preceptor llega al final de la lista) App Móvil → API Backend: GET /api/reportes?page=2&limit=20&search={texto}.
  11. (Pasos 3–6 se repiten para la página 2, y la App añade los resultados a la lista existente).

4.5.5 Secuencia: Crear Reporte (Preceptor)

  1. Preceptor → App Móvil: En "Reportes", presiona "+ Nuevo Reporte".
  2. App Móvil → App Móvil: Navega a CrearReportePreceptorScreen.
  3. Preceptor → App Móvil: Ingresa matrícula del estudiante, escribe motivo, presiona "Guardar".
  4. App Móvil → App Móvil: Lee claveEmpleado del preceptor (reportadoPor) del UserProvider.
  5. App Móvil → API Backend: POST /api/reportes/crear con { matriculaReportado, reportadoPor, tipoUsuarioReportante: 'Preceptor', motivo }.
  6. (Pasos 11–15 como en "Monitor: Crear Reporte").

4.5.6 Secuencia: Ver Todas las Amonestaciones

  1. Preceptor → App Móvil: Navega a "Amonestaciones".
  2. App Móvil → API Backend: GET /api/amonestaciones.
  3. API Backend → Base de Datos: SELECT A.*, E.NombreCompleto, P.NombreCompleto, N.Nombre FROM Amonestaciones A JOIN Estudiantes E JOIN Preceptores P JOIN Cat_NivelAmonestacion N ....
  4. Base de Datos → API Backend: Devuelve lista completa de amonestaciones.
  5. API Backend → App Móvil: Respuesta 200 OK con { success: true, data: [...] }.
  6. App Móvil → Preceptor: Muestra la lista de amonestaciones.

4.5.7 Secuencia: Crear Amonestación

  1. Preceptor → App Móvil: En "Amonestaciones", presiona "+ Nueva Amonestación".
  2. App Móvil → App Móvil: Navega a CrearAmonestacionScreen.
  3. App Móvil → API Backend: GET /api/amonestaciones/niveles (para dropdown de nivel).
  4. (API obtiene niveles de BD y los devuelve a la App).
  5. App Móvil → Preceptor: Muestra formulario (Matrícula, Nivel, Motivo).
  6. Preceptor → App Móvil: Ingresa datos, presiona "Guardar".
  7. App Móvil → App Móvil: Lee claveEmpleado del preceptor del UserProvider.
  8. App Móvil → API Backend: POST /api/amonestaciones/registrar con { matriculaEstudiante, clavePreceptor, idNivel, motivo }.
  9. (API guarda en BD y devuelve éxito a la App).
  10. App Móvil → Preceptor: Muestra mensaje de éxito.
  11. App Móvil → App Móvil: Regresa a la lista, opcionalmente recarga.

4.5.8 Secuencia: Ver Estado de Limpieza General

  1. Preceptor → App Móvil: Navega a "Limpieza".
  2. App Móvil → API Backend: GET /api/limpieza/cuartos-con-calificacion.
  3. (Pasos 3–7 como en "Monitor: Ver Lista de Cuartos").
  4. App Móvil → Preceptor: Muestra la lista de todos los cuartos agrupados por pasillo, con su última calificación.
